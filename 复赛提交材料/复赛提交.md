### 1. Demo 展示

**视频链接**：（待上传/填写，建议 3-5 分钟，包含以下三段）

**演示流程概述**：

1. **会前 proactive 自动触发**
   - 在飞书日历创建未来 2 小时内的会议（如"飞书 AI 大赛内部讨论会"）
   - OpenClaw 定时任务每小时自动扫描主日历（`window_minutes=120`）
   - `calendar_watcher.py` 成功识别到该事件，自动检索历史相关文档和未完成的 action items
   - `briefing_writer.py` 生成会前简报（飞书文档），包含会议概览、相关背景资料、历史待办联动
   - **消息推送**：通过飞书 IM 自动推送会前简报卡片给参会人（已验证端到端打通）
   - **产物**：飞书文档（会前简报）+ 飞书消息提醒

2. **会后 companion 处理**
   - 用户向 Agent（B）发送会议纪要文档链接或 docx_token
   - Agent 调用 `openclaw-meeting-companion` skill，进入 `agent.py` 处理流程
   - **双 Tier 策略提取 action items**：
     - Tier 1（P0）：优先解析飞书智能纪要的 `<checkbox>` 标签，直接读取 AI 已生成的待办
     - Tier 2（P1）：无 checkbox 时，基于规则+NLP fallback 提取正文中的行动项
   - `normalizer.py` 补齐负责人、截止时间、状态，输出结构化 ActionItem
   - `base_writer.py` 自动创建/复用 Base 表格，upsert 行动项记录
   - `distribution_writer.py` 生成可直接转发到群聊的分发稿文档
   - **产物**：Base 追踪表 + 分发稿文档 + 负责人执行清单

3. **跟进与复用**
   - 再次运行同一文档时，自动更新已有记录（基于任务内容哈希匹配），避免重复插入
   - 历史 action items 可关联到后续会议的会前简报，形成知识闭环

### 2. 核心代码展示

**项目仓库**：`openclaw-meeting-agent/`

**关键模块说明**：

| 模块 | 文件 | 职责 |
|---|---|---|
| 统一调度 | `scheduled_runner.py` | 定时轮询，串联 proactive 触发器（会前简报 + 纪要扫描 + 队列处理） |
| 日历扫描 | `src/calendar_watcher.py` | 扫描未来 120 分钟内即将开始的会议，兼容 HTTP API 和 lark-cli 两种返回格式 |
| 会前简报 | `src/briefing_writer.py` | 基于相关文档和历史任务生成会前简报，自动创建飞书文档 |
| 智能纪要解析 | `src/smart_minutes_parser.py` | 解析飞书 AI 生成的智能纪要（`<checkbox>` 标签），准确率最高路径 |
| 行动项提取 | `src/action_extractor.py` | 基于规则和 NLP 从会议纪要中抽取 action items |
| 标准化 | `src/normalizer.py` | 补齐负责人、截止时间、状态，输出结构化 ActionItem |
| Base 写入 | `src/base_writer.py` | 创建/复用 Base 表格，upsert action items，支持重试和幂等 |
| 分发稿生成 | `src/distribution_writer.py` | 生成可直接转发的飞书文档（Markdown 渲染） |
| 消息推送 | `src/feishu_client.py` | 封装飞书 IM 消息发送，自动处理 union_id → open_id 转换 |
| 纪要扫描 | `src/minutes_watcher.py` | 扫描新增会议纪要，触发会后处理，支持去重 |
| OpenClaw 封装 | `openclaw-skill/scripts/run.py` | OpenClaw Skill 入口，无参自动 fallback 到 scheduler 模式 |

**鉴权架构（Triple-backend）**：

```
user_token (OpenClaw JWT)
    ↓ 优先
lark-cli (本地开发，浏览器授权)
    ↓ fallback
tenant_token (app_id + app_secret，HTTP API)
```

- `src/feishu_client.py` 统一封装三种鉴权模式，自动探测、自动 fallback，确保本地开发和 OpenClaw 线上环境都能稳定运行，无需人工干预鉴权切换。

**已解决的典型工程问题**：
1. **日历事件扫描失败**：lark-cli `calendar +agenda` 返回格式与 HTTP API 不一致（无 `event_id`、时间格式为 ISO 字符串而非 timestamp），通过 `_parse_iso_ts()` 和多重格式兼容解决
2. **跨应用消息发送**：全局 union_id 无法直接发送消息，通过 `contact/v3/users/batch_get_id` 自动转换为 app-scoped open_id
3. **定时任务无参调用**：OpenClaw 定时任务默认不带参数，`run.py` 自动识别无参场景并 fallback 到 `scheduler --auto --once` 模式
4. **Base 管理 API 兼容性**：JWT 环境下 Base list/create 接口行为差异，通过 graceful fallback + 手动 `--base-token` 绕过

### 3. 项目亮点介绍

**方向 B：会议与项目的全链路伴侣**

本 Agent 围绕"开会"这一企业最高频场景，构建了**会前-会后-跟进**的完整闭环：

**会前**：基于日历事件自动触发（支持定时扫描），检索历史相关文档和未完成的 action items，生成高密度会前简报。解决"参会者背景信息缺口"和"会议准备不足"的痛点。

**会后**：自动解析会议纪要（优先读取飞书 AI 智能纪要的 checkbox 待办，无 checkbox 时规则兜底），抽取结构化 action items，补齐负责人、截止时间、状态，写入飞书 Base 追踪表，同时生成可直接转发的分发稿和负责人执行清单。

**跟进**：所有 action items 沉淀到统一 Base，支持重复运行时自动更新而非重复插入，历史待办可联动到后续会议的会前简报，形成团队可追踪、可分派、可对账的项目推进入口。

**区别于通用问答 Bot**：本 Agent 不回答"会议讲了什么"，而是直接产出**可执行、可追踪、可分发的知识资产**。

**主动触发能力**：
- **定时触发**：`scheduled_runner.py` 每小时自动扫描日历（会前简报）和会议纪要（会后处理），无需用户对话即可自动运行
- **手动触发**：`meeting_companion.py` / `agent.py` 支持用户指定 docx ID 直接运行完整链路

### 4. AI 亮点介绍

#### 4.1 项目中使用了哪些高阶 AI 技巧？

**智能分层解析（双 Tier 策略）**

针对飞书智能纪要和人工撰写的会议纪要，采用"置信度分层"策略，而非一刀切地依赖大模型：

- **Tier 1（P0 - 智能纪要优先）**：直接解析飞书 AI 生成的 `<checkbox>` 标签，提取 AI 已识别的待办事项。这是最高置信度路径，几乎零幻觉，准确率接近 100%。**技巧点**：不重复造轮子，直接复用飞书 AI 已经理解并标注好的结构化信息。
- **Tier 2（P1 - 规则兜底）**：当 Tier 1 返回为空时，基于规则+NLP fallback 提取正文中的行动项。通过正则+关键词+句式模板（如"XXX 负责 YYY，Z 日之前完成"）做局部语义匹配。**技巧点**：在低成本和高覆盖之间取平衡，不调用 LLM 也能处理 80% 以上的常见场景。
- **Tier 3（P2 - LLM 增强，预留）**：规则覆盖不足时，可接入大模型 API 做深层语义理解。

**知识关联与自动补齐**

- **背景链接自动关联**：为每个 action item 自动检索并关联相关背景文档链接，降低执行者的信息断层。
- **历史待办联动**：在会前简报中提示"与本次会议相关的未完成待办"，帮助参会者理解上下文，避免重复讨论。

#### 4.2 项目中人和 AI 的分工是怎么样的？

| 分工 | AI（Agent） | 人类 |
|---|---|---|
| **信息提取** | 从纪要中抽取任务、负责人、截止时间 | 复核提取结果，纠正歧义 |
| **知识关联** | 自动搜索相关文档、关联历史待办 | 判断关联是否准确，补充遗漏背景 |
| **格式化输出** | 生成 Base 表格、分发稿、执行清单 | 微调措辞，确认分发范围 |
| **状态追踪** | 自动更新已有记录，避免重复插入 | 定期 review Base，推动执行 |
| **消息推送** | 自动发送会前简报和提醒 | 关注消息，准时参会 |

**关键设计**：对于负责人缺失或时间模糊的条目，Agent **不猜测**，而是标记为"待确认"状态，推送给组织者人工复核。这降低了幻觉风险，也符合"AI 做重复劳动，人类做判断决策"的实际工作流。

#### 4.3 项目中包含了哪些核心模型选型思路？

**"规则优先，模型兜底"的选型原则**

本项目没有盲目追求"全用大模型"，而是根据任务的确定性程度选择最合适的工具：

| 任务类型 | 选型 | 理由 |
|---|---|---|
| Checkbox 提取（确定性高） | **规则（XPath/正则）** | 飞书已用 AI 标注好，直接读取零成本、零幻觉 |
| 固定句式提取（确定性中） | **规则+NLP** | "负责...截止..."等句式模式稳定，正则+关键词足够覆盖，速度快、成本低 |
| 模糊语义理解（确定性低） | **LLM（预留）** | 仅在规则失效时启用，避免过度依赖大模型带来的延迟和费用 |

**长文本 vs 结构化输出的定位**

- **长文本理解**：利用大模型的长上下文能力（200k+），一次性读取完整会议纪要，避免分段导致的上下文丢失。
- **结构化输出**：不依赖模型的通用对话能力，而是将其定位为"结构化信息提取器"，输出格式严格对应 Base 字段（任务、负责人、截止时间、状态、来源会议、背景知识、相关链接），便于直接写入数据库。

#### 4.4 引入 AI 后对原有工作流带来了哪些改变？

**Before（人工方式）**：
1. 会前：参会者手动翻找历史文档和待办，准备不充分，会议效率低
2. 会后：会议组织者花 15-20 分钟手动整理 action items，复制粘贴到 Excel/文档
3. 分发：手动排版分发稿，发到群里经常遗漏关键信息
4. 跟进：散落在多个文档和聊天记录里，容易遗忘，无人对账

**After（AI Agent 方式）**：
1. **会前**：Agent 自动扫描日历，提前 2 小时生成简报并推送，参会者零准备成本
2. **会后**：Agent 1-2 分钟自动提取 action items、补齐信息、写入 Base，效率提升 ~90%
3. **分发**：自动生成标准化分发稿，格式统一、信息完整，可直接转发
4. **跟进**：所有 action items 沉淀到统一 Base，支持自动更新和历史联动，形成可追踪的推进入口

**核心价值**：把"开会"从信息黑洞变成**可追踪、可分发的协作资产**，让会议真正产生执行力。
