本地跑通了"会前简报 + 会后行动项追踪"完整链路。从飞书文档读取 → 智能纪要解析 →行动项提取 → 多维表格写入 → 分发稿生成，全部在本地用 lark-cli 调通。

### 二、量化指标 （必填）

- 完成 12 个 Python 模块的开发（document_reader、action_extractor、base_writer、distribution_writer、smart_minutes_parser、briefing_writer、normalizer 等）

- 输出 1 份 OpenClaw Skill 初版文档（SKILL.md + scripts/run.py）


### 三、过程复盘与沉淀（必填）

#### 1. 这周主要搞定了哪些具体环节？用了什么方法或工具辅助？

- 需求拆解：把赛题"会议全链路伴侣"拆成"会前简报"和"会后追踪"两个独立阶段，再拆出文档读取、提取、规范化、Base 写入、消息推送 5 个模块。

- Base 幂等写入：实现了"已存在则更新，不存在则创建"的逻辑，避免重复跑脚本时数据乱掉。
- AI 分工：先用了豆包2.0pro，后面又用了GPT5.4和kimi_v2.6帮我写模块骨架和异常处理，我自己调 API参数和边界逻辑（比如日期正则、标题去噪）。

#### 2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？

- 卡点1：智能纪要检测总是返回 False。排查后发现 document_reader.py 里的 _normalize_document_content() 把 HTML 标签全 strip 了，导致 <checkbox> 没了。解法：给 DocumentRecord 加了一个 raw_content 字段，保留原始 HTML，检测时读 raw_content。
- 卡点2：OpenClaw 安装 skill 后跑的是内置工具，不是我自己写的脚本。发现是 skill 路径写了 {baseDir}/../meeting_companion.py，但文件被复制到了 skill 根目录。解法：统一改成 {baseDir}/meeting_companion.py。

#### 3. 有没有什么你觉得这次写得特别顺，或者下次还能直接复用的东西？

- 三层降级提取策略：P0（飞书智能纪要 <checkbox>）→ P1（规则提取"待办"区块）→ P2（LLM兜底）。这个结构以后做任何文档提取都能套。
- 幂等 Base 写入模式：先 find_record_id_by_task 查重，再决定 create 还是update，做数据同步类任务很稳。


### 四、本周随手记（选填）

- lark-cli 的 docs +fetch 返回的是原始 HTML，不是 markdown，这是飞书 docx和普通文档最大的区别。
- search:docs:read 这个 scope 很容易被漏掉，但 pre-meeting 简报极度依赖它。
- OpenClaw 的 {baseDir} 变量指向的是 skill压缩包解压后的根目录，所有相对路径都要基于它来算。

## 第二周期（4.26-4.28）

### 一、核心产出 （必填）

把整个会议伴侣的触发逻辑由用户”手动使用"升级为"自动触发"，并解决了在飞书OpenClaw云端平台上无法运行的问题。现在系统不需要用户说话，自己定时扫描日历，会议开始前主动推送简报；发现新的会议纪要也会自动处理。同时在本地和云端两套环境里都能跑通。

### 二、量化指标 （必填）

- 新增 4 个核心能力（自动扫描日历、自动发现纪要、统一调度器、云端双环境适配）
- 覆盖 10个以上飞书接口（文档、表格、日历、消息、通讯录）
- 修复 5 个以上接口适配问题（表格字段格式、日历命令、文档内容获取等）
- 最终成功提取 8 条行动项全部写入飞书表格，零丢失
- 本地 + 云端双环境验证通过

### 三、过程复盘与沉淀（必填）

#### 1. 这周主要搞定了哪些具体环节？用了什么方法或工具辅助？

- 自动触发架构：实现了三种自动触发方式——①扫描用户日历，会议开始前30分钟自动发会前简报；②自动发现新上传的会议纪要并提取任务；③定时处理队列里的待办文档。这是赛题方向 B的核心要求。
- 云端适配：OpenClaw 平台里没有浏览器，无法像本地那样扫码登录飞书。解决：通过解密用户登录令牌，直接调用飞书API，云端和本地用的是同一个身份，能访问的数据完全一样。
- 双环境兼容：同一套业务逻辑，底层自动判断——本地用飞书官方 CLI 工具（功能最全），云端直接调 HTTP接口（无需安装），代码层面完全无感切换。


#### 2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？

- 坑 1：云端拿不到智能纪要的待办内容。飞书官方 API 返回的文档内容里，AI生成的勾选框数据被过滤掉了。解决：先用OpenClaw 内置的文档抓取工具拿到完整内容，再通过参数注入传给脚本处理，绕过官方 API 的限制。
- 坑2：同一个表格接口，云端和本地返回的数据格式不一样。本地CLI 工具返回的字段名叫 name，云端 HTTP 接口叫field_name；字段类型一个用中文一个用数字。解决：给两套接口分别写适配层，对外暴露统一的业务接口。
- 坑 3：以为有"列出日历事件"的命令，实际不存在。查帮助文档才发现飞书 CLI里这个命令叫"查看日程"，参数名也不一样。


#### 3. 有没有什么你觉得这次写得特别顺，或者下次还能直接复用的东西？

- 双后端设计：同一套业务代码，底层自动切换本地/云端，以后做任何需要"本地开发+云端部署"的飞书应用都能直接复用。
- 外部预处理+内容注入：当官方 API覆盖不到某些数据时（比如智能纪要的勾选框），先用工具抓取完整内容再注入处理，保证功能不降级。

### 四、本周随手记（选填）

- urllib（Python 标准库）比第三方 HTTP 库更适合做云端fallback，因为沙箱里不一定让装额外依赖。
- 自动触发才是赛题方向 B 的得分核心，"对话式调用"只是入口之一，这次从"被动响应"改成"主动推送"是关键架构修正。
- 飞书多维表格的接口在不同调用方式下字段命名不统一，做跨平台适配时最容易踩坑，需要时刻对照官方文档。

## 第三周期（4.29-5.1）

### 一、核心产出 （必填）

会前主动推送链路端到端跑通，解决了日历扫描在多环境下的格式兼容性、消息跨应用身份转换、以及重复触发等问题，实现了"创建日历事件 → 自动扫描 → 生成简报 → 推送参会人"的完整闭环。

### 二、量化指标 （必填）

- 修复日历扫描 3 个兼容性根因，覆盖 lark-cli 与 HTTP API 双模式
- 实现会议事件去重机制，避免定时任务重复推送
- 打通 union_id → open_id 跨应用消息发送链路
- 会前简报扫描窗口从 30 分钟扩展至 120 分钟，匹配赛题要求
- 本地环境消息推送成功率达到 100%

### 三、过程复盘与沉淀（必填）

#### 1. 这周主要搞定了哪些具体环节？用了什么方法或工具辅助？

- 日历扫描鲁棒性：lark-cli 返回的日历数据与 HTTP API 格式差异很大，缺少 event_id、时间字段为 ISO 字符串或嵌套字典。通过手写兼容解析逻辑，统一归一化为标准时间对象，兼容 Python 3.10。
- 消息身份转换：飞书全局 union_id 无法直接跨应用发消息，通过通讯录接口批量转换为应用内 open_id，失败时自动降级原样发送。
- 事件去重：扫描窗口扩大到 120 分钟后，同一事件可能被多次扫到。通过事件唯一键持久化到本地文件，实现跨进程去重。

#### 2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？

- 坑 1：日历扫描"静默失败"。lark-cli 返回的日历事件没有 event_id 字段，原代码直接 KeyError 后被异常捕获吞掉，日志无报错。解决：把必填过滤改为兼容判断，允许 lark-cli 格式通过。
- 坑 2：Python 3.10 的 `fromisoformat` 不支持带冒号的时区偏移（如 +08:00），解析直接报错。解决：用正则预处理时区格式，再统一走 `strptime` 解析。
- 坑 3：消息发出后收件人显示为自己发给自己，而非应用 Bot。因为 user_token 模式下发消息用的是用户身份。解决：检测到 user_token 时，独立换取应用的 tenant_access_token，以 Bot 身份推送。

#### 3. 有没有什么你觉得这次写得特别顺，或者下次还能直接复用的东西？

- 多格式时间兼容解析器：以后处理飞书任何时间字段（timestamp、ISO 字符串、嵌套字典）都能直接套用。
- 事件去重模板：JSON 持久化 + 唯一键生成，做任何定时扫描类任务都能复制。
- 身份自动切换：user_token / lark-cli / tenant_token 三层 fallback 加上 Bot 身份特殊处理，以后写任何跨环境飞书工具都能复用。

### 四、本周随手记（选填）

- lark-cli 发消息的实际 flag 是 `--user-id`，和官方文档写的 `--receive-id` 不一致。
- 去重状态必须持久化到磁盘，OpenClaw 每次定时任务都是新进程，内存变量会丢失。
- `im:message:send_as_user` scope 容易被漏授权，重新登录时务必检查权限列表。

## 第四周期（5.2-5.4）

### 一、核心产出 （必填）

OpenClaw 云端端到端验证通过。解决了 Skill 无参调用、代码路径错误、云端 lark-cli 缺失导致的全链路 fallback 问题，定时任务能在沙箱里稳定运行并自动推送简报。

### 二、量化指标 （必填）

- Skill 版本从 1.001 迭代到 1.009，累计修复 6 个云端专属问题
- run.py 新增无参自动 fallback，支持定时任务零配置进入调度模式
- 修复 Skill 代码路径错误，解决云端"跑旧代码"问题
- Triple-backend 鉴权完成全链路验证：JWT → lark-cli → tenant_token 自动降级
- 完成 3 轮部署-测试-修复循环，端到端消息发送云端验证通过

### 三、过程复盘与沉淀（必填）

#### 1. 这周主要搞定了哪些具体环节？用了什么方法或工具辅助？

- 无参 fallback：OpenClaw 定时任务默认不带参数，Agent 无法判断该调哪个脚本。在 run.py 里增加无参自动判定逻辑，无输入时直接进入调度器模式。
- 代码路径修复：skill 解压后的目录结构与本地不同，原路径计算少了一层 parent，导致云端 import 的是旧副本。修正路径并清理冗余文件。
- Bot token 独立获取：user_token 模式下消息会显示为用户自发自收。在消息发送模块里增加独立换取 tenant_access_token 的逻辑，以应用 Bot 身份推送。
- zip 打包优化：排除 tmp、缓存和旧 zip 文件，减小体积，避免旧文件干扰。

#### 2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？

- 坑 1：上传新版本 zip 后，定时任务跑的仍是旧逻辑。排查发现是代码路径错误导致加载了 skill 目录内的旧副本。解决：修正 project_root 路径，打包时排除冗余副本。
- 坑 2：Agent 在"运行助手任务"模式下偏好使用内置工具而非调用 skill，强制提示词反而导致死锁。workaround：依赖定时任务直接执行 run.py，绕开 Agent 意图判断。
- 坑 3：Base 管理接口在 JWT 环境下返回格式与 HTTP API 不一致，自动发现 Base 失败。解决：增加 graceful fallback，允许手动传入 base-token，保证核心写入不中断。

#### 3. 有没有什么你觉得这次写得特别顺，或者下次还能直接复用的东西？

- 无参 fallback 模式：以后写任何需要定时自动运行的 OpenClaw skill，都可以套用"无参数自动进调度器"的设计。
- Bot 独立鉴权片段：用标准库直接请求飞书 auth 端点换取 token，不依赖统一封装，适合任何需要"以应用身份发消息"的场景。
- 云端部署循环：改路径 → 精简 zip → 上传 → 看日志 → 修复，这个循环跑熟后，云端调试效率大幅提升。

### 四、本周随手记（选填）

- OpenClaw 日志有 1-2 分钟延迟，不能指望实时刷新。
- 飞书 auth 接口的返回结构和其他接口不同，token 在根层级而非 data 字段内，这是唯一不能走统一封装的接口。
- cron 时区没有北京/上海，选 UTC 或东京后需手动换算北京时间。
