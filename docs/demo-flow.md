# Demo 流程设计

## 1. 目标

本 Demo 的目标是把已经手工验证过的流程，固化为一条可重复执行的命令行工作流：

```bash
python agent.py --docx <飞书文档ID>
```

执行完成后，程序应自动完成：

1. 读取飞书会议纪要文档。
2. 抽取结构化行动项。
3. 创建或复用飞书 Base。
4. 写入行动项记录。
5. 生成飞书分发稿文档。
6. 输出 Base 链接和分发稿链接。

## 2. Demo 边界

第一版只处理单篇会议纪要文档，不做批量处理。

第一版只支持用户显式传入 `docx` 文档 ID，不做自动监听和定时扫描。

第一版只写入 Base 和分发稿，不自动发群消息。

## 3. 主流程

### Step 1: 输入解析

程序入口接收以下参数：

- `--docx`: 飞书文档 ID，必填。
- `--base-name`: Base 名称，可选。
- `--table-name`: 表名称，可选。

这一层只负责参数校验和任务启动，不处理业务逻辑。

### Step 2: 读取文档

调用飞书文档读取接口，根据 `docx` 文档 ID 获取正文纯文本。

输出结果：

- 文档标题
- 文档正文
- 文档链接

失败条件：

- 文档不存在
- 没有读取权限
- 返回正文为空

### Step 3: 抽取行动项

从会议纪要正文中提取行动项，生成标准化结构。

第一版建议优先使用规则抽取，不强依赖额外模型：

- 优先扫描 `待办`、`Action Items`、`后续工作计划` 等标题块。
- 识别任务描述、负责人、截止时间、状态。
- 未识别到的字段置空或标记为 `需确认`。

输出结果为内存中的行动项列表。

### Step 4: 规范化字段

对抽取结果做统一处理：

- 去除多余空白和换行。
- 将截止时间标准化为日期对象或毫秒时间戳。
- 补齐默认状态。
- 追加来源会议和来源文档链接。
- 将未识别负责人或日期的任务标记为 `需确认`。

### Step 5: 准备 Base

检查是否存在可用 Base。

第一版建议采用以下策略：

1. 若用户未指定 Base，则创建新 Base。
2. 在 Base 中创建固定名称的行动项表。
3. 若表已存在，则直接复用。

输出结果：

- `app_token`
- `table_id`
- Base 链接

### Step 6: 写入行动项

将规范化后的行动项逐条写入 Base。

写入策略：

- 一条行动项对应一条 Base 记录。
- 每条记录写入统一字段。
- 失败时保留错误信息，方便定位哪条记录写入失败。

输出结果：

- 成功写入的记录数
- 失败记录列表

### Step 7: 生成分发稿

基于行动项列表生成 Markdown 文本，并调用飞书文档导入接口创建分发稿。

分发稿至少包含：

- 来源会议信息
- 原始文档链接
- Base 链接
- 行动项汇总表
- 每条任务的背景说明

输出结果：

- 分发稿文档链接

### Step 8: 汇总输出

程序执行结束后，在终端输出：

- 处理的文档标题
- 抽取到的行动项数量
- Base 链接
- 分发稿链接
- 是否存在需确认字段

## 4. 模块拆分

建议拆成以下模块：

- `agent.py`
  - 命令行入口，负责串联流程。
- `src/feishu_client.py`
  - 封装飞书文档、Base、文档导入等操作。
- `src/document_reader.py`
  - 负责读取并包装会议纪要文档。
- `src/action_extractor.py`
  - 负责从正文中抽取行动项。
- `src/normalizer.py`
  - 负责字段标准化和默认值填充。
- `src/base_writer.py`
  - 负责 Base 创建、表准备和记录写入。
- `src/distribution_writer.py`
  - 负责生成分发稿 Markdown 并创建飞书文档。
- `src/models.py`
  - 定义统一的数据结构。

## 5. 推荐执行顺序

代码开发时按以下顺序实现最稳：

1. 实现数据结构。
2. 实现文档读取。
3. 实现规则版行动项抽取。
4. 实现 Base 创建和写入。
5. 实现分发稿生成。
6. 在 `agent.py` 中串起来。

## 6. 第一版成功标准

满足以下条件即可视为第一版 Demo 可用：

1. 输入一篇飞书会议纪要文档 ID。
2. 程序能稳定读取正文。
3. 至少抽取出主要行动项。
4. 能成功写入飞书 Base。
5. 能生成飞书分发稿文档。
6. 终端输出最终链接。

## 7. 后续扩展点

后续可以沿这条流程扩展：

- 增加模型抽取，提升负责人和截止时间识别准确率。
- 接入 Wiki 搜索，为任务补充背景知识链接。
- 支持增量更新而不是重复写入。
- 增加定时触发。
- 增加消息卡片分发。
