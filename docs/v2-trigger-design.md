# V2 主动触发实现设计

## 1. 目标

在当前 V1 会后闭环基础上，补齐比赛要求中的“至少一种主动触发方式”。

V2 不追求复杂度，目标是：

- 自动发现待处理会议纪要
- 自动执行现有抽取与分发流程
- 自动生成处理结果
- 保留运行日志和状态

## 2. 选择的触发方式

V2 优先实现：

- 定时触发

原因：

- 实现成本最低
- 对现有代码侵入最小
- 演示方式清晰
- 足够满足比赛的主动触发要求

## 3. 触发场景

建议定义一个“待处理会议纪要池”，Agent 周期性扫描并自动处理。

可选实现方式：

- 方案 A：本地配置文件维护待处理文档列表
- 方案 B：用一个飞书 Base 作为“待处理文档清单”
- 方案 C：指定一个文档目录或搜索关键词，定时搜索新纪要

V2 最建议先做：

- 方案 B：飞书 Base 驱动

原因：

- 所有输入都留在飞书里
- 演示效果最好
- 状态天然可视化

## 4. 建议的输入表设计

可以新建一个 Base 或在已有 Base 中加一张表，例如：

- 表名：`待处理会议纪要`

字段建议：

- `文档链接`
- `文档ID`
- `会议主题`
- `处理状态`
- `上次处理时间`
- `结果Base链接`
- `结果分发稿链接`
- `备注`

其中 `处理状态` 可用值：

- `待处理`
- `处理中`
- `已完成`
- `失败`

## 5. V2 流程

定时任务每轮执行：

1. 读取 `待处理会议纪要` 表中状态为 `待处理` 的记录
2. 提取文档 ID
3. 调用当前 `agent.py` 主流程
4. 成功后回写：
   - `处理状态 = 已完成`
   - `上次处理时间`
   - `结果Base链接`
   - `结果分发稿链接`
5. 失败后回写：
   - `处理状态 = 失败`
   - `备注 = 错误信息`

## 6. 代码层改造建议

### 新增文件建议

- `src/queue_reader.py`
  - 读取待处理文档列表
- `src/job_runner.py`
  - 执行单条任务并汇总结果
- `scheduled_runner.py`
  - 定时入口

### 现有代码复用

当前主流程尽量不拆散，复用：

- [agent.py](../agent.py)
  - 已有 `run()` 可以直接复用
- [src/feishu_client.py](../src/feishu_client.py)
  - 继续扩展读写“待处理会议纪要”表的能力

### 需要新增的方法

建议在 `src/feishu_client.py` 增加：

- `list_pending_documents(...)`
- `mark_job_processing(...)`
- `mark_job_done(...)`
- `mark_job_failed(...)`

## 7. 调度方式

本地演示版可用两种：

- 方式 A：循环 sleep
- 方式 B：Windows 计划任务 / cron

V2 建议先做：

- `scheduled_runner.py --once`
- `scheduled_runner.py --loop --interval 300`

这样既能：

- 单次运行便于调试
- 循环运行便于演示“主动触发”

## 8. 命令行设计建议

### 单次扫描

```bash
python scheduled_runner.py --once
```

### 循环扫描

```bash
python scheduled_runner.py --loop --interval 300
```

### 指定输入表

```bash
python scheduled_runner.py --queue-base-token <base_token> --queue-table-name 待处理会议纪要 --once
```

## 9. 演示方式

比赛演示时可以这样讲：

1. 在飞书里准备一条状态为 `待处理` 的纪要记录
2. 启动定时 Runner
3. Runner 自动读取并处理文档
4. Base 中状态自动变成 `已完成`
5. 自动回填结果链接
6. 展示结果 Base 和分发稿

这个演示比“我手动运行一个脚本”更符合赛题要求。

## 10. V2 交付标准

V2 完成后，至少满足：

- 存在一种主动触发方式
- 不需要手动把文档 ID 逐条喂给 `agent.py`
- 能自动处理待处理任务
- 能把处理结果写回飞书

## 11. 已知风险

### 风险 1：重复处理

解决：

- 使用 `处理状态`
- 用文档 ID 做幂等标识

### 风险 2：处理中断

解决：

- 增加 `处理中` 状态
- 失败时回写错误信息

### 风险 3：规则抽取误差

解决：

- 将 `Needs confirmation` 暴露给用户
- 保留原始背景说明，便于人工校验

## 12. 推荐的下一步实现顺序

1. 先建 `待处理会议纪要` 输入表
2. 新增读队列和回写状态的客户端方法
3. 新增 `scheduled_runner.py`
4. 跑通 `--once`
5. 再补 `--loop`

## 13. 结论

V2 不需要一开始就做复杂事件监听。

对当前比赛阶段最合理的策略是：

> 先用“飞书输入表 + 定时扫描”的方式把主动触发做出来，满足赛题要求并提升项目完整度，再继续扩展会前背景推送与项目对账能力。
