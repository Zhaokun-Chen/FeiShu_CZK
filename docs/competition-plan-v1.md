# 比赛版项目方案 V1

## 1. 项目名称

会议与项目的全链路伴侣 Agent

## 2. 赛题选择

- 赛道：飞书 OpenClaw 赛道
- 课题：企业办公知识整合与分发 Agent
- 方向：方向 B，会议与项目的全链路伴侣（偏协作与对齐）

## 3. 一句话定义

围绕企业会议场景，Agent 在会前主动推送背景知识，在会后自动抽取行动项并补齐负责人、截止时间和背景链接，持续沉淀到统一追踪表，并生成可直接分发的知识产物，帮助团队完成从“信息对齐”到“任务推进”的闭环。

## 4. 为什么要做这个

企业里真正高价值的信息并不缺，缺的是：

- 会前不知道应该看哪些历史文档
- 会中虽然讨论很多，但行动项容易被埋在长纪要里
- 会后虽然有纪要，但负责人、截止时间、背景依据不统一
- 后续推进缺少统一入口，任务容易分散在文档、群聊和口头同步中

传统搜索或问答只能解决“我知道要搜什么”的问题，不能解决：

- 什么时候该主动推送什么信息
- 如何把会前背景、会后任务和后续推进串起来
- 如何把非结构化会议内容变成可追踪、可分发、可协作的资产

## 5. 目标用户

- 项目负责人
- 会议组织者
- 参会成员
- 团队协作者
- 需要跨文档快速理解项目上下文的人

## 6. 核心价值

这个项目希望带来的价值不是“更会答问题”，而是：

- 会前：减少参会者对项目背景的信息缺口
- 会后：减少行动项遗漏和二次人工整理
- 跟进：将任务沉淀到统一表格，形成推进入口
- 分发：把原始会议内容转成高密度知识产物，而不是原样堆给用户

## 7. 产品形态

这个 Agent 不是单纯的聊天对话框，而是多种知识产物的生成器：

- 会前背景 briefing
- 会后行动项追踪 Base
- 行动项分发稿文档
- 后续可扩展成：
  - 飞书消息卡片
  - 群推送摘要
  - CLI 终端提示
  - 周报 / 风险洞察

## 8. 版本规划

### V1：会后闭环 MVP

当前已做到的能力：

- 读取飞书会议纪要文档
- 抽取 Action Items
- 补齐负责人、截止时间、状态、背景说明
- 写入飞书 Base
- 生成飞书分发稿
- 重复运行时更新已有记录

这是当前仓库已经验证过的最小闭环。

### V2：主动触发版

目标：

- 满足赛题“至少一种主动触发方式”的要求

建议优先实现：

- 定时触发

预期能力：

- 定时扫描待处理会议纪要
- 识别新增或未处理文档
- 自动执行 V1 流程
- 自动生成最新追踪结果和分发稿

### V3：会前背景知识推送

目标：

- 真正补齐方向 B 的“全链路”属性

预期能力：

- 输入会议主题或会前文档
- 自动检索历史会议、相关文档、相关待办
- 生成高密度会前 briefing
- 分发给会议组织者或参会人

### V4：项目推进与自动对账

目标：

- 从“会后整理工具”升级为“项目推进中枢”

预期能力：

- 聚合多场会议中的相关任务
- 与统一 Base 追踪表对账
- 同步状态、阻塞原因和背景链接
- 识别长期未推进事项并预警

## 9. 主动触发策略

赛题要求至少一种主动触发方式。当前建议优先选：

- 定时触发

原因：

- 最容易稳定交付
- 不依赖复杂 webhook 或群事件监听
- 便于比赛演示

后续可扩展：

- 事件触发
  - 指定文档创建后自动处理
  - 指定群里发送会议纪要链接后自动处理
- 阈值触发
  - 当会议纪要中待办超过一定数量时自动生成重点清单

## 10. 效果验证思路

比赛里必须自证效果，建议从三类指标设计：

### 准确性

- Action Item 抽取准确率
- 负责人识别准确率
- 截止时间识别准确率
- 背景说明是否可追溯到原文

### 效率

- 人工整理会议行动项平均耗时
- Agent 生成行动项清单耗时
- 会后分发准备时间缩短比例

### 接受度

- 用户是否愿意直接使用生成稿
- 是否愿意以生成 Base 作为后续跟踪入口
- 人工修改次数或修改比例

## 11. 比赛答辩时的核心卖点

- 不是通用问答，而是高密度知识产物生成
- 不是纯总结，而是落到真实飞书协作资产
- 不是被动搜索，而是主动触发和主动分发
- 不是单点会议工具，而是能向项目推进中枢演进

## 12. 当前阶段结论

当前仓库最准确的定位不是“完整比赛项目”，而是：

> 方向 B 的会后子场景 MVP，已经验证了真实飞书闭环，可以作为比赛版全链路 Agent 的核心基础模块。
