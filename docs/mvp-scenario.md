# MVP 场景定义：会后行动项与知识分发 Agent

## 1. 场景定位

本项目面向飞书 OpenClaw 赛道“企业办公知识整合与分发 Agent”，第一版选择“会议与项目的全链路伴侣”方向，聚焦会后纪要场景。

MVP 定位为：

> 用户提供一篇飞书妙记或会议纪要文档后，Agent 自动读取正文，抽取 Action Items，补齐负责人、截止时间、来源依据和背景说明，写入飞书 Base 行动项追踪表，并生成一篇可直接转发的会议行动项分发稿。

## 2. 目标用户

- 项目负责人
- 会议组织者
- 参会成员
- 需要跟进会议行动项的团队协作者

## 3. 核心问题

企业会议结束后，会议纪要里通常包含大量讨论背景、关键决策和待办事项，但行动项容易被埋在长篇妙记中。

常见问题包括：

- Action Items 分散在会议总结、待办、章节记录和关键决策中。
- 负责人、截止时间、背景依据不统一。
- 会后分发依赖人工整理，容易遗漏。
- 任务缺少统一追踪入口，后续推进成本高。

## 4. Agent 要完成的任务

第一版 Agent 完成以下闭环：

1. 读取飞书妙记或会议纪要文档。
2. 从正文中抽取 Action Items。
3. 标准化每条行动项字段。
4. 补充来源会议、背景依据和相关说明。
5. 写入飞书 Base 行动项追踪表。
6. 生成飞书分发稿文档。
7. 返回 Base 链接和分发稿链接，供用户确认后分发。

## 5. 输入与输出

### 输入

- 飞书文档链接或文档 ID。
- 文档内容通常来自飞书妙记总结、会议纪要或人工整理的会议文档。

### 输出

- 飞书 Base 行动项追踪表。
- 飞书行动项分发稿文档。
- 每条行动项包含以下字段：
  - 任务
  - 负责人
  - 截止时间
  - 状态
  - 来源会议
  - 背景知识 / 来源依据

## 6. 使用的飞书能力

MVP 阶段使用以下飞书能力：

- 飞书文档读取：读取会议纪要正文。
- 飞书 Base 创建：创建行动项追踪库。
- 飞书 Base 表和记录写入：保存结构化行动项。
- 飞书 Base 查询：校验任务可追踪。
- 飞书文档生成：生成可分发的行动项文档。
- 群聊列表读取：仅用于识别可见项目群，不在 MVP 中自动群发。

## 7. 为什么普通搜索 / 问答不够

普通搜索要求用户自己知道要搜什么、去哪里搜，并手动判断哪些内容可执行。

普通问答只能回答单次问题，不能把会议内容沉淀成可追踪、可分发、可协作的工作资产。

本 Agent 的价值在于：

- 将非结构化会议纪要转化为结构化任务。
- 为每条任务保留来源依据，降低幻觉和误解风险。
- 自动生成可分发文档，减少会后人工整理时间。
- 通过 Base 形成统一追踪入口，支持后续状态维护和项目推进。

## 8. 第一版边界

MVP 暂不做以下能力：

- 不自动群发消息。
- 不读取外部群成员。
- 不直接创建飞书任务。
- 不做复杂权限管理。
- 不做全公司级知识库检索。
- 不做长期自动对账和预警。

这些能力可作为后续版本扩展。

## 9. 已验证的最小闭环

当前已通过 MCP 实际验证以下流程：

1. 读取飞书妙记生成的飞书文档。
2. 抽取会议行动项。
3. 创建飞书 Base。
4. 创建行动项追踪表。
5. 写入 6 条行动项记录。
6. 查询 Base 记录并校验可读回。
7. 生成飞书行动项分发稿文档。

## 10. 下一步

下一步将把已验证的手工 MCP 调用固化为本地可运行 Demo：

```bash
python agent.py --docx <飞书文档ID>
```

目标是让命令自动完成：

1. 读取文档。
2. 抽取行动项。
3. 写入 Base。
4. 生成分发稿。
5. 输出结果链接。

为保证演示稳定性，当前版本额外优化为：

- 支持直接输入飞书文档链接或文档 ID
- 支持复用已有 Base，减少重复创建资源
- 对相同任务优先更新已有记录，降低演示污染
- 本地导出 JSON 报告，便于回看抽取结果
